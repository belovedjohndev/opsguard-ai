# Request creation API and verified tenant context

**Roadmap slice:** Week 1, Day 6 — Fastify API and Tenant Context  
**Status:** Accepted implementation specification  
**Date:** 2026-07-18

## Purpose

This document defines the first HTTP and PostgreSQL adapter slice for OpsGuard AI. It exposes only
liveness and authenticated request creation, invokes the existing `CreateRequest` application use
case, verifies tenant membership before constructing application context, and persists the request,
initial status history, and creation audit event atomically.

The development/test authentication stub is not production authentication. It exists only to prove
the tenant-context, authorization, transport, application, and persistence boundaries before a
production identity provider is selected.

## HTTP surface

### `GET /health`

Success response:

```json
{
  "status": "ok"
}
```

The endpoint is unauthenticated and reports process liveness only. It does not query PostgreSQL or
expose configuration, credentials, dependency versions, stack traces, or readiness details.

### `POST /v1/requests`

Accepted JSON body:

```json
{
  "sourceType": "form",
  "sourceReference": "form-submission-42"
}
```

Both fields are required strings. The transport schema rejects malformed JSON, missing values, and
obvious type/size violations. It accepts only the Day 4 source vocabulary and a source reference of
at most 255 characters. The domain remains authoritative for request invariants.

The schema uses `additionalProperties: false` with Fastify's configured removal of additional
properties. A supplied `tenantId`, `membershipId`, `userId`, `requestId`, `status`, timestamp, audit
field, or workflow field is removed before the handler maps the body to the use case. In particular,
a body `tenantId` can never override authenticated context.

Success response (`201 Created`):

```json
{
  "requestId": "018f47d2-68df-7a8b-9c01-23456789abcf",
  "tenantId": "018f47d2-68df-7a8b-9c01-23456789abcd",
  "status": "received",
  "createdAt": "2026-07-18T09:00:00.000Z"
}
```

The response contains only the application result. It never contains database rows, membership
details, audit metadata, SQL information, or internal error values.

No other request route is introduced on Day 6.

## Fastify application factory

`buildApp(dependencies)` constructs and returns one independent Fastify instance. It:

- receives the `CreateRequest` use case and active-membership resolver through explicit injection;
- owns route, request-context, request-ID, schema, and error-handler registration;
- opens no database connection and reads no process environment;
- can be created, injected, and closed independently in tests; and
- has no model, workflow, provider, storage, Redis, MinIO, or Temporal dependency.

Production startup is a separate composition root. It validates configuration, creates the
application PostgreSQL connection and adapters, checks connectivity before listening, and closes the
Fastify instance and database pool during shutdown. Startup reports a deterministic generic failure
without printing credentials or connection URLs.

## Request correlation IDs

Fastify receives a `genReqId` function that returns a server-generated UUID for every request.
`requestIdHeader` remains disabled, so an arbitrary client value cannot become authoritative. The
correlation ID is separate from the domain `RequestId` generated by `CreateRequest`.

Every response includes the authoritative ID in `x-request-id`. Every error body repeats the same
value as `requestId`. Tests validate UUID syntax, header/body equality on errors, and rejection of a
client attempt to choose the correlation ID.

Full OpenTelemetry correlation is deferred to Day 13.

## Development/test authentication stub

Protected requests must supply:

- `x-opsguard-user-id`; and
- `x-opsguard-tenant-id`.

These synthetic headers are untrusted. The stub validates both as canonical, non-zero UUIDs. It does
not accept a membership ID. It then queries PostgreSQL for exactly one membership matching both user
and tenant, requiring:

- `tenant_memberships.status = active`; and
- `tenants.status = active`.

The membership ID and role come only from that query. The Day 4 `users` table has no user lifecycle
status, so inactive-user rejection is not representable yet; membership and tenant suspension are
enforced.

Missing or malformed identity returns `401`. A well-formed user/tenant pair without an active
membership and active tenant returns `403`. Database unavailability returns `503`; an unexpected
resolver failure returns `500`. Responses do not reveal whether a tenant, user, suspended tenant, or
suspended membership exists.

The stub implements no passwords, JWTs, OAuth, API keys, sessions, registration, or identity-provider
integration and must not be enabled as production authentication without replacement.

## Tenant selection and immutable context

After verification, the request receives a frozen context containing only:

- branded `UserId`;
- branded `TenantId`;
- server-resolved branded `TenantMembershipId`;
- verified membership role; and
- the server-owned correlation ID.

The Fastify request decorator is initialized with `null` and assigned a newly frozen value per
request. No credentials or raw header values are retained. Protected handlers fail closed if context
is absent and never reconstruct tenant identity from the request body.

`CreateRequest` receives `tenantId` and `actorMembershipId` only from this verified context.

## Authorization policy

Authentication identifies the presented user. Membership resolution proves active participation in
the selected active tenant. Authorization then applies the Day 6 policy.

No accepted document defines a narrower request-creation role restriction, so every active Day 4
membership role may create a request:

- `owner`;
- `operations_manager`;
- `reviewer`;
- `operator`; and
- `auditor`.

This deliberately minimal policy is explicit and isolated. It does not create a general RBAC
framework. A later restriction requires an accepted policy change and focused tests.

## Input validation

Fastify/AJV owns transport shape validation and sanitization. The route constructs only the existing
`CreateRequestCommand`. The domain continues to own source vocabulary, nonblank source identity,
initial status, and request invariants. The route does not assign request IDs, status, timestamps,
history, audit facts, or persistence values.

Malformed JSON and invalid route bodies return the same safe `400` envelope; raw AJV issue arrays
and parser messages are not returned.

## Application persistence contract evolution

Day 5 made request and initial status history one atomic repository payload. Day 6 extends
`CreateRequestPersistence` with one application-owned `request.created` audit draft. The use case
constructs the draft from already validated domain facts:

- tenant ID;
- actor membership ID;
- entity type `request`;
- request ID;
- occurrence timestamp equal to creation time;
- metadata containing only `status` and `sourceType`.

This makes the audit side effect explicit in the application contract. It avoids a hidden adapter
side effect, a best-effort post-commit audit call, and a generic Unit of Work abstraction.

## PostgreSQL adapter and transaction boundary

The `@opsguard/database` adapter implements the application-owned repository interface. It converts
the request snapshot, initial status value, and audit draft into Drizzle inserts. Every tenant-owned
insert explicitly supplies `tenant_id`.

One Drizzle/PostgreSQL transaction inserts, in order:

1. the `requests` row;
2. the initial `request_status_history` row; and
3. the `audit_events` row.

The transaction commits all three or none. Routes never open transactions and never import Drizzle.
The adapter uses only the application PostgreSQL connection and never reads Temporal settings except
the existing guard that prevents selecting the Temporal database name.

## Audit event

The creation event is:

```json
{
  "eventType": "request.created",
  "entityType": "request",
  "metadata": {
    "status": "received",
    "sourceType": "form"
  }
}
```

Tenant ID, request entity ID, actor membership ID, and occurrence timestamp are stored in their
dedicated columns. Metadata excludes the source reference, request body, authentication headers,
credentials, SQL details, stack traces, customer messages, and future model data.

No audit update/delete trigger is introduced.

## Transaction failure and database error mapping

Any insert failure rejects the transaction and rolls back earlier inserts. Integration tests install
a temporary trigger only inside a guarded isolated database to force the audit insert to fail, then
prove request, history, and audit rows are all absent.

The adapter inspects PostgreSQL errors internally:

- SQLSTATE `23505` maps to `CONFLICT` only when the named constraint is
  `requests_tenant_id_source_key`;
- connection-class and shutdown/connection-capacity failures map to `UNAVAILABLE`; and
- every other failure maps to `UNEXPECTED`.

SQLSTATE, table names, constraint names, queries, credentials, and stack traces do not cross the
adapter boundary.

## HTTP error mapping

One transport mapper produces stable envelopes:

| Condition | Status | Public code |
|---|---:|---|
| Malformed JSON or invalid route body | 400 | `INVALID_REQUEST` |
| Invalid application request input | 400 | `INVALID_REQUEST` |
| Missing or malformed authentication headers | 401 | `AUTHENTICATION_REQUIRED` |
| No active tenant membership | 403 | `TENANT_ACCESS_DENIED` |
| Tenant-scoped source-reference conflict | 409 | `REQUEST_CONFLICT` |
| Membership or request persistence unavailable | 503 | `SERVICE_UNAVAILABLE` |
| Unexpected resolver, application, or persistence failure | 500 | `INTERNAL_ERROR` |

Example:

```json
{
  "error": {
    "code": "REQUEST_CONFLICT",
    "message": "A request with this source reference already exists."
  },
  "requestId": "2f515100-58b4-4c6f-94ca-dbe1f5031200"
}
```

The mapper does not reveal whether another tenant owns a matching record. Unexpected details remain
internal.

## Safe logging and responses

Request bodies and authentication headers are never logged by application code. Internal diagnostics
contain only the correlation ID and a stable failure category. Responses contain no stack, SQL,
constraint, connection, environment, or dependency details. Startup errors are generic.

## Test strategy

Fastify component tests use `inject()` with fakes and close every application instance. They cover
health, response request IDs, client request-ID rejection, malformed JSON, body validation, missing
and malformed authentication, missing active membership, body tenant spoofing, application error
mapping, and internal-detail redaction.

PostgreSQL integration tests create random databases matching the existing guarded
`opsguard_test_<32 lowercase hex>` pattern, migrate from empty, seed two tenants and memberships, and
drop only that database. They verify persisted rows and tenant scopes, missing context, tenant body
spoofing, membership spoofing, duplicates, and transaction rollback using row counts and constraints.
The normal application database is never reset or deleted.

Normal `pnpm test` remains database-free. A separate mandatory API integration command runs in CI
against the existing application PostgreSQL service.

## Explicit exclusions

Day 6 does not implement:

- production authentication, login, passwords, JWTs, OAuth, sessions, or API keys;
- RLS or general RBAC administration;
- request listing, retrieval, update, deletion, or transition routes;
- the Day 7 expanded threat model;
- model gateways, providers, prompts, or structured assessment;
- Temporal workflows, Redis queues, MinIO storage, or external integrations;
- React UI, documents, retrieval, or evaluations;
- OpenTelemetry instrumentation, rate limiting, webhook intake, or deployment automation.
