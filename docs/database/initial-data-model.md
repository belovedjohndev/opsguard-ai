# Initial tenant-aware data model

**Roadmap slice:** Week 1, Day 4 — Initial Data Model and Migrations
**Status:** Accepted implementation specification
**Date:** 2026-07-18

## Purpose

This specification defines the first PostgreSQL persistence boundary for OpsGuard AI. It
establishes database-enforced tenant ownership, stable identities, minimal lifecycle metadata,
model-run metadata, and request audit history before repositories or application behavior exist.

The schema belongs to the `@opsguard/database` infrastructure package. It must not define domain
state-transition policy, authorization, HTTP behavior, provider adapters, or workflow execution.

## Table model

All primary keys are PostgreSQL UUIDs generated by the database. Every tenant-owned table also has
a named unique constraint on `(tenant_id, id)` so child tables can use composite foreign keys that
carry the same tenant identifier.

### `tenants`

Purpose: stable organization identity and coarse lifecycle state.

- Primary key: `id`.
- Required identity: normalized, globally unique `slug` and human-readable `name`.
- Lifecycle: `status` using `tenant_status`.
- Timestamps: `created_at`, `updated_at`.
- The slug is lowercase and limited to letters, digits, and single hyphen-separated segments.

### `users`

Purpose: platform user identity independent from any tenant membership. Authentication-provider
accounts and credentials are deliberately absent.

- Primary key: `id`.
- Required identity: normalized, globally unique `email`.
- Timestamps: `created_at`, `updated_at`.
- Email is stored lowercase and trimmed. Authentication and email verification remain deferred.

### `tenant_memberships`

Purpose: assign one platform user a role and lifecycle state inside one tenant.

- Primary key: `id`.
- Ownership: `tenant_id` references `tenants.id`.
- User: `user_id` references `users.id`.
- Required role and status use constrained PostgreSQL enums.
- Uniqueness: `(tenant_id, user_id)` prevents duplicate membership; `(tenant_id, id)` supports
  tenant-safe child references.
- Timestamps: `created_at`, `updated_at`.

### `requests`

Purpose: persistence record for an operational request aggregate without implementing its Day 5
state machine.

- Primary key: `id`.
- Ownership: `tenant_id` references `tenants.id`.
- Source identity: `source_type` plus non-empty `source_reference` is unique per tenant. This is the
  durable, tenant-scoped intake identity required for later deduplication.
- Optional creator: `created_by_membership_id` is absent for verified non-human sources such as
  email and webhooks. When present, `(tenant_id, created_by_membership_id)` must reference a
  membership in the same tenant.
- Current status uses the minimal `request_status` enum.
- Uniqueness: `(tenant_id, id)` and `(tenant_id, source_type, source_reference)`.
- Timestamps: `created_at`, `updated_at`.

### `request_status_history`

Purpose: append-oriented recording of the initial request status and later status changes. It does
not authorize or execute transitions.

- Primary key: `id`.
- Ownership and request relationship: `(tenant_id, request_id)` references the same-tenant request.
- `is_initial` distinguishes the first entry. Initial entries require `previous_status IS NULL`;
  later entries require a non-null previous status different from `next_status`.
- A partial unique index permits only one initial entry per request.
- Optional actor: `changed_by_membership_id` is absent for system or verified external-source
  changes. When present, its composite foreign key enforces same-tenant membership.
- Timestamp: `changed_at`.

### `prompt_versions`

Purpose: immutable, tenant-owned prompt-version metadata without prompt execution logic or raw
prompt logging.

- Primary key: `id`.
- Ownership: `tenant_id` references `tenants.id`.
- Identity: normalized `prompt_key`, positive integer `version`, and lowercase SHA-256
  `content_sha256` identifying the separately managed prompt content.
- Uniqueness: `(tenant_id, id)` and `(tenant_id, prompt_key, version)`.
- Timestamp: `created_at`.

Prompt versions are tenant-scoped because tenant policy and knowledge influence prompt behavior.
System-wide prompt distribution, activation, and content storage are later decisions.

### `model_configurations`

Purpose: tenant-owned, non-secret model selection metadata.

- Primary key: `id`.
- Ownership: `tenant_id` references `tenants.id`.
- Identity: normalized `configuration_key`, `provider`, and `model` identifiers.
- Uniqueness: `(tenant_id, id)` and `(tenant_id, configuration_key)`.
- Timestamp: `created_at`.

API keys, credentials, provider SDK configuration, arbitrary parameters, activation policy, budgets,
and routing behavior are not stored in this table on Day 4.

### `ai_runs`

Purpose: model-execution metadata only; it cannot authorize a request decision or persist model
payloads.

- Primary key: `id`.
- Ownership and request: `(tenant_id, request_id)` references a same-tenant request.
- Prompt and model selection: composite foreign keys require the prompt version and model
  configuration to belong to the same tenant.
- Lifecycle: `status` using `ai_run_status`.
- Optional provider/result metadata: `provider_request_id`, input/output token counts, latency, error
  classification, `started_at`, and `completed_at`. Absence means the provider has not supplied the
  value or the lifecycle stage does not use it.
- Token counts and latency cannot be negative. Completion cannot precede start.
- A non-null provider request identifier is unique within its tenant and model configuration.
- Timestamp: `created_at`.

No prompts, customer message bodies, model responses, credentials, or unrestricted payloads are
stored.

### `audit_events`

Purpose: append-oriented material-event evidence for tenant requests.

- Primary key: `id`.
- Ownership: `tenant_id` references `tenants.id`.
- Optional actor: `(tenant_id, actor_membership_id)` references a same-tenant membership when a
  tenant member caused the event. Null means a system or verified external-source actor.
- Entity: Day 4 permits only `entity_type = 'request'`, and `(tenant_id, entity_id)` references the
  same-tenant request. This avoids an unenforceable polymorphic tenant relationship.
- `event_type` is a normalized event name. Domain event vocabulary remains application-owned.
- `metadata` must be a JSON object no larger than 16 KiB when serialized. It may contain only
  redacted, non-secret identifiers and decision metadata—never credentials, complete prompts,
  customer message bodies, or unrestricted PII.
- Timestamp: `occurred_at`.

Database triggers preventing update or delete are deferred. Day 4 uses restrictive foreign keys and
an append-only contract; later database roles and application authorization must deny mutation in
normal operation. Corrections add events rather than rewriting existing events.

## Status enums

- `tenant_status`: `active`, `suspended`.
- `membership_role`: `owner`, `operations_manager`, `reviewer`, `operator`, `auditor`.
- `membership_status`: `active`, `suspended`.
- `request_source_type`: `form`, `email`, `webhook`, `service_account`.
- `request_status`: `received`, `assessing`, `needs_information`, `pending_review`, `rejected`,
  `completed`, `failed`.
- `ai_run_status`: `queued`, `running`, `succeeded`, `failed`, `cancelled`.

The request statuses are persistence vocabulary derived from the accepted intake, assessment,
information-request, review, rejection, completion, and explicit-failure flows. They do not define
legal transitions; Day 5 owns that state machine and may require a forward migration if its accepted
vocabulary differs.

## Tenant-boundary invariants

1. Tenant identity is never accepted from model output as authority.
2. Every tenant-owned row has a direct foreign key to `tenants`.
3. Every tenant-owned parent has a unique `(tenant_id, id)` key.
4. Request creator, status-history request and actor, AI-run request/prompt/model, and audit
   entity/actor relationships use composite foreign keys containing `tenant_id`.
5. Supplying tenant B with a tenant A identifier must fail at the PostgreSQL boundary even if the
   referenced UUID exists.
6. RLS is intentionally absent on Day 4. Day 6 authenticated tenant context must complement, not
   replace, these relational constraints.

## Timestamp policy

- Persisted timestamps use PostgreSQL `timestamptz` with millisecond precision.
- Database defaults set creation, occurrence, and change timestamps with `now()`.
- `updated_at` exists only on mutable identity/current-state rows. Later transactional application
  writes must set it explicitly; Day 4 does not add hidden update triggers.
- All application and test interpretation uses UTC. PostgreSQL retains an absolute instant.

## Deletion behavior

All foreign keys use explicit restrictive delete and update behavior. There are no cascading deletes
or silent orphaning. Tenant, user, membership, request, prompt, and model-configuration deletion is
blocked while dependent evidence exists. Lifecycle enums are used instead of physical deletion for
the initial mutable records. Retention, erasure, anonymization, and legal deletion workflows require
a later specification and migration.

## Index and uniqueness policy

- Primary keys provide direct UUID lookup.
- Every foreign-key access path receives an explicit index unless an existing unique index has the
  same leading columns.
- Tenant request queues index `(tenant_id, status, created_at)`.
- Request history indexes `(tenant_id, request_id, changed_at)`.
- AI runs index `(tenant_id, request_id, created_at)`.
- Audit lookup indexes `(tenant_id, entity_type, entity_id, occurred_at)` and tenant chronology.
- Constraint and index names are explicit and stable so integration tests can assert deterministic
  PostgreSQL failures.

## Migration strategy

- Drizzle TypeScript schema is the code-first source used to generate checked-in SQL and metadata.
- Migrations are append-only, ordered, reviewed artifacts. CI and local verification apply them to a
  completely empty isolated PostgreSQL database.
- Migration application uses only the OpsGuard application PostgreSQL connection settings. It never
  connects to or changes Temporal persistence.
- The initial migration creates enums, tables in dependency order, constraints, and indexes. It does
  not create application data.
- The pgvector image remains the PostgreSQL baseline, but the `vector` extension is not enabled
  until a retrieval migration introduces a concrete vector column and indexing design.
- Direct schema push is not an accepted migration path.

## Rollback and recovery

Generated Drizzle migrations are forward-only in this repository; no automatic destructive down
migration is claimed.

- Integration tests create randomly named `opsguard_test_*` databases, apply all migrations from
  empty state, and drop only that guarded prefix during cleanup.
- Local developers may discard only an explicitly isolated test database. The normal
  `APP_POSTGRES_DATABASE` is never reset by database tests.
- Production rollback uses a verified backup/point-in-time restore when reverting persisted data is
  required, or a reviewed compensating/forward-fix migration when data can be preserved.
- Later destructive or data-transforming migrations require backups, preflight validation, explicit
  transaction/locking analysis, and a rehearsed recovery procedure before deployment.

## Deliberately deferred assumptions

- Day 5 domain entities, branded IDs, transition policy, repositories, use cases, and transaction
  boundaries.
- Day 6 authentication, authorization, authenticated tenant context, database roles, and RLS
  evaluation.
- Service-account identity tables and authentication-provider mappings.
- Request payload/body storage, documents, embeddings, pgvector extension/indexes, and retrieval.
- Prompt content storage, activation/supersession policy, model-provider adapters, and model calls.
- Approval, proposal, integration execution, outbox/inbox, workflow, reconciliation, budget, and
  evaluation tables.
- Audit immutability triggers, partitioning, retention, erasure/anonymization, and cryptographic
  integrity controls.
- Production deployment credentials, pooling, replicas, high availability, and backup automation.
